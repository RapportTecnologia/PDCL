Ao criar qualquer arquivo markdown neste projeto, siga obrigatoriamente estas regras de formatação. Inicie sempre com badges específicos substituindo `<org>` e `<repository>` pelos valores corretos do projeto: badge de visitantes, licença CC BY-SA 4.0, idioma português, tecnologias utilizadas (Python 3.8+, Jupyter Notebook, Machine Learning), status do projeto, tamanho do repositório e data do último commit. Imediatamente após os badges, inclua o header animado com a API capsule-render usando as cores padrão do projeto (0:0f172a,50:1a56db,100:10b981) e texto personalizado conforme o conteúdo do arquivo. O conteúdo principal deve usar títulos hierárquicos começando por ## para seções principais, ### para subseções e #### para detalhamento, evitando o uso de # que deve ser reservado apenas para títulos principais quando estritamente necessário. Formate todo código usando blocos com sintaxe highlighting específica da linguagem e código inline com crases. Centralize todas as imagens usando tags `<p align="center">` com atributos alt descritivos. Use links absolutos para recursos externos e relativos para arquivos internos, sempre no formato [texto](link) padronizado. Antes do resumo final, inclua o footer animado com cores invertidas (0:10b981,50:1a56db,100:0f172a). O resumo final é obrigatório e deve conter data de criação, autor, versão, última atualização, atualizado por e histórico detalhado de alterações com formato específico. O projeto utiliza ferramenta automática markdown_history_manager.py integrada com git hooks para gerenciar histórico de alterações, configurada em .markdown_history_config.json e executada automaticamente ao fazer commit de arquivos .md. Ao criar ou modificar arquivos, verifique sistematicamente: badges presentes e corretos, header animado incluído, footer animado incluído, resumo final preenchido, estrutura hierárquica correta e formatação consistente. Indente corretamente todas as listas e subitens mantendo a hierarquia visual clara. Use linguagem portuguesa em toda documentação e mantenha consistência terminológica em todo o projeto. Para exemplos de código, use sempre blocos completos com sintaxe highlighting e inclua explicações detalhadas quando necessário. Documente todas as decisões de design e arquitetura usando seções específicas com exemplos práticos e justificativas claras. Mantenha todos os arquivos de documentação atualizados com as últimas alterações do projeto e revise periodicamente para garantir relevância e precisão das informações apresentadas.
